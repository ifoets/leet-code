8.	What is false sharing? How to avoid it?
    False sharing occurs when multiple threads modify independent variables that reside on the same CPU cache line, causing unnecessary cache coherency traffic and performance degradation.
    Key Points
    Threads do not share the variable logically, but share the cache line physically.
    Every write invalidates the cache line in other cores.
    Leads to high latency, low throughput, and CPU contention.
    Cache line (64 bytes)
    | varA | varB | varC | varD |

    Thread 1 → updates varA
    Thread 2 → updates varB

    ❌ Both threads cause cache invalidation even though they modify different variables
    2️⃣ How to Avoid False Sharing
    Use @Contended (Java 8+)
     @Contended
        public volatile long counter1;

    Key Principle
    A properly constructed object in Java guarantees that:
    All final fields are visible to other threads after construction.
    Non-final fields may not be visible unless published safely.

    class User {
        final String name;
        final int age;

        User(String name, int age) {
            this.name = name;
            this.age = age;
        }
    }
    Any thread that sees a User object will see name and age correctly initialized.
    🔹 2. Non-Final Fields
    Non-final fields are not guaranteed to be visible to other threads unless you use:
    volatile
    synchronized
    Atomic*
    Safe publication

3.	Why CMS was removed?
    CMS (Concurrent Mark Sweep) was removed because it could no longer meet modern JVM requirements for predictable latency, large heaps, and operational simplicity.
4.	Explain G1 regions and pause prediction.
    1️⃣ Why G1 Introduced Regions
    Old collectors:
    •	Young = contiguous
    •	Old = contiguous
    •	Fragmentation & unpredictable pauses
    G1 solution:
    👉 Break the heap into equal-sized regions.
    ________________________________________
    2️⃣ G1 Heap Layout (Region-Based)
    Heap
     ├── Region 1  (Young - Eden)
     ├── Region 2  (Young - Eden)
     ├── Region 3  (Survivor)
     ├── Region 4  (Old)
     ├── Region 5  (Old)
     ├── Region 6  (Humongous)
    Region size
    •	Power of 2
    •	Typically 1MB–32MB
    •	Chosen at JVM startup based on heap size
    📌 ~2000 regions is the target.
    ________________________________________
    3️⃣ Region Types (Dynamic!)
    Region Type	Purpose
    Eden	New allocations
    Survivor	Young survivors
    Old	Long-lived objects
    Humongous	Large objects (>50% of region)
    ⚠️ Regions can change roles dynamically.
    ________________________________________
    4️⃣ Young GC in G1
    Trigger
    •	Eden regions fill up
    What happens
    1.	STW pause
    2.	Copy live objects:
    o	Eden → Survivor
    o	Survivor → Survivor / Old
    3.	Update region roles
    4.	Free entire Eden regions
    📌 Still a copying GC, but region-based.
    ________________________________________
    5️⃣ Mixed GC (G1’s Big Innovation)
    Instead of:
    •	Only Young
    •	Or Full Old GC
    G1 does:
    Young + selected Old regions together
    Which Old regions?
    ➡ Regions with most garbage
    This is why it’s called Garbage First.

5.	What is SATB (Snapshot-At-The-Beginning)?
    SATB (Snapshot-At-The-Beginning) is a concurrent marking technique used by modern JVM garbage collectors (notably G1, CMS, Shenandoah) to correctly find live objects while the application is still running.
    Think of it as:
    “Pretend the heap is frozen at the start of marking, even though the app keeps changing it.”
    ________________________________________
    Why SATB Is Needed
    During concurrent marking, the application threads (mutators) continue to:
    •	Create new objects
    •	Update references
    •	Remove references
    Without protection, the GC could:
    ❌ Miss live objects
    ❌ Accidentally reclaim reachable objects


6.	ZGC vs G1 – when would you choose ZGC?
    Core Difference (One Line)
    •	G1 → Best-effort pause targets, region-based, balanced
    •	ZGC → Pause time is the priority, heap size is almost irrelevant
    ________________________________________
    When ZGC Is the Right Choice ✅
    1️⃣ You Have Strict Latency SLAs
    If your service has:
    •	p99 / p999 latency budgets
            p99 / p999 are latency percentiles — they tell you how slow the slowest requests are, not the average.
            This is extremely important for microservices, GC tuning, and SRE discussions.
            ________________________________________
            Simple Definition
            •	p99 = 99th percentile latency
            👉 99% of requests are faster than this value
            👉 1% are slower
            •	p999 = 99.9th percentile latency
            👉 99.9% of requests are faster
            👉 0.1% are slower (1 in 1000 requests)

    •	User-facing APIs
    •	Real-time or near–real-time requirements
    Example
    p99 must be < 50 ms
    G1 pauses (50–200 ms) → ❌ risky
    ZGC pauses (~1–5 ms) → ✅ safe
    ________________________________________
    2️⃣ Heap Size Is Large (≥ 8–16 GB)
    G1 pause time grows with live set size.
    ZGC:
    •	Pause time is nearly independent of heap size
    •	Works well with tens or hundreds of GB
    Signal
    Increasing heap → G1 pauses get longer
    ➡ Switch to ZGC
    ________________________________________
    3️⃣ High Allocation Rate + Long-Lived Objects
    •	Reactive / async services
    •	Object-heavy frameworks
    •	JSON / serialization heavy paths
    G1:
    •	Promotion pressure
    •	Mixed GC becomes aggressive
    ZGC:
    •	Concurrent relocation
    •	No evacuation pauses tied to region count


7.	What causes Full GC even with G1?
    1️⃣ Promotion Failure (Most Common)
    What happens
    •	Objects survive Young / Mixed GC
    •	JVM tries to promote them to Old regions
    •	Not enough contiguous free regions
    ➡ G1 triggers Full GC (STW, single-threaded compaction)
    Why this happens
    •	Old Gen almost full
    •	High allocation rate
    •	Survivor spaces too small
    📌 Very common under memory pressure.
    ________________________________________
    2️⃣ To-Space Exhaustion (G1-Specific)
    What it is
    During evacuation:
    •	Objects are copied to To-space
    •	To-space runs out mid-GC
    ➡ G1 cannot continue evacuation
    Result
    •	Immediate Full GC
    📌 This is unique to G1 and very important.
    ________________________________________
    3️⃣ Humongous Object Pressure
    Reminder
    •	Object > 50% of region size
    •	Allocated directly in Old Gen
    •	Requires contiguous regions
    Problems
    •	Fragmentation
    •	Regions get pinned
    •	Hard to reclaim incrementally
    ➡ G1 falls back to Full GC
    📌 Large byte arrays, buffers, caches are common causes.
    ________________________________________
    4️⃣ Metadata / Metaspace Exhaustion
    If:
    •	Metaspace becomes full
    •	Class unloading can’t free enough space
    ➡ Full GC is triggered
    6️⃣ Allocation Failure During GC
    If JVM cannot allocate:
    •	Eden
    •	Survivor
    •	Old regions
    Even after mixed collections
    ➡ Full GC is the last resort.
    ________________________________________
    7️⃣ JVM Startup / Ergonomics Edge Cases
    •	Very small heaps
    •	Bad region sizing
    •	Extreme fragmentation
    G1 may decide:
    “Full GC is cheaper than incremental cleanup”


8.	What is GC root?
    GC Roots are the starting points from which the JVM determines object reachability during garbage collection.
    In simple terms:
    If an object is reachable from a GC root, it is considered alive.
    If it’s not reachable from any GC root, it’s eligible for GC.
    ________________________________________
    Why GC Roots Exist (Core Idea)
    The JVM does not guess which objects are alive.
    It uses a reachability graph:
    GC Roots → reachable objects → reachable objects → ...

    Common Types of GC Roots (Very Important)
    1️⃣ Thread Stack References (Most Common)
    2️⃣ Static Fields
    3️⃣ JNI (Native) References
    4️⃣ Class Metadata (Metaspace)
    5️⃣ Synchronization Locks
    6️⃣ Finalizer Queue References
    Role of GC Roots in Modern GCs
    •	Used at marking start
    •	Combined with:
    o	SATB (G1, CMS)
    o	Load barriers (ZGC)
    They define the initial live set.

9.	How to debug memory leaks in production?
    Golden Rule ⚠️
    Observe first. Dump later. Restart last.
    Never start by taking heap dumps blindly.
    ________________________________________
    1️⃣ Detect: Is It Really a Memory Leak?
    Memory leak pattern
    Used heap ↑↑↑
    GC runs
    Used heap does NOT come down
    Not a leak
    •	Heap grows, then stabilizes
    •	GC recovers memory periodically
    ________________________________________
    Tools (Safe in prod)
    •	Prometheus + Grafana
    •	JVM metrics (jvm_memory_used_bytes)
    •	GC logs
    Look for:
    •	Old Gen steadily growing
    •	Full GC not reclaiming memory
    ________________________________________
    2️⃣ Enable GC Logs (Low Risk)
    -Xlog:gc*,gc+heap=info,gc+age=trace
    What to check:
    •	Old Gen after Full GC
    •	Promotion failures
    •	Humongous allocations
    •	GC frequency increasing
    📌 If Old Gen after Full GC keeps increasing → leak confirmed.


10.	Difference between soft, weak, phantom references.
    1️⃣ SoftReference
    What it means
    •	Object is kept as long as memory is sufficient
    •	Cleared before OutOfMemoryError
    GC behavior
    •	GC collects it only under memory pressure
    •	May survive many GC cycles
    Typical use case
    ✔ Caches
    ✔ Memory-sensitive data
    2️⃣ WeakReference
    What it means
    •	Object is collected as soon as it’s weakly reachable
    •	Cleared on next GC cycle
    GC behavior
    •	GC does not care about memory pressure
    •	Survives no GC
    Typical use case
    ✔ Metadata caches
    ✔ Map keys (WeakHashMap)
    ✔ Listener references
    3️⃣ PhantomReference
    What it means
    •	Object is already unreachable
    •	get() always returns null
    GC behavior
    •	Enqueued after finalization
    •	Used to track object reclamation
    Typical use case
    ✔ Native resource cleanup
    ✔ Off-heap memory
    ✔ Cleaner / DirectByteBuffer

4.	What is CAS? ABA problem?
    CAS is an atomic instruction that updates a memory location only if it has an expected value, otherwise it fails.
    Concept
    CAS(address, expectedValue, newValue)
    Steps:
    1.	Read memory at address
    2.	Compare it with expectedValue
    3.	If equal → update to newValue atomically
    4.	Else → fail (retry usually)

15.	How to design a lock-free algorithm?
    2️⃣ Key Principles of Lock-Free Design
    1.	Atomic operations
    o	Use hardware atomic instructions (via java.util.concurrent.atomic.*)
    o	E.g., compareAndSet (CAS), getAndAdd, getAndSet
    2.	Avoid shared mutable state
    o	Minimize the amount of data multiple threads write
    3.	Retry loops (spin loops)
    o	If CAS fails (another thread updated the value), retry until success
    4.	Immutability when possible
    o	Makes reasoning easier and avoids partial updates
    ________________________________________
    3️⃣ Steps to Design a Lock-Free Algorithm
    Step 1: Identify the shared resource
    •	E.g., a counter, stack, queue, linked list
    Step 2: Define the atomic operation
    •	Decide what state change needs to be atomic
    •	Example: increment counter
    Step 3: Use CAS (Compare-And-Swap)
    AtomicInteger counter = new AtomicInteger(0);

    int oldValue, newValue;
    do {
        oldValue = counter.get();
        newValue = oldValue + 1;
    } while (!counter.compareAndSet(oldValue, newValue));
    •	Ensures no locks
    •	Thread retries if another thread wins the race
    ________________________________________
    Step 4: Handle contention
    •	Contention causes retries → use backoff strategies
    •	Example:
    Thread.sleep(random(1, 10));
    ________________________________________
    Step 5: Avoid ABA problem (if needed)
    •	CAS can fail if a variable changes from A → B → A
    •	Use:
    o	AtomicStampedReference (version number)
    o	AtomicMarkableReference (boolean flag)
